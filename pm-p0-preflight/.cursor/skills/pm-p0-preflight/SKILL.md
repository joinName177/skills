# PMflow P0 澄清预审

帮用户在进入 `/pm-flow` 正式流程**之前或并行**，把 P0 澄清问题「问对、答全、可粘贴」，并确保**用户截图、批注、选项选择、@文件内容**都能被正确吸收进草拟回答。

当需求涉及**入口、导航、列表、弹框、抽屉、面板、表单、按钮、筛选、切换、移动端/Web 差异**时，必须额外用交互视角审查：不要只问“做什么”，还要问清“用户如何理解、如何进入、如何返回、哪里会被打断、哪里会犯错”。

## 何时使用

| 输入 | 动作 |
|------|------|
| 仅有一句话需求 | 生成标准 P0 问题 + 草拟回答 |
| 已有 pm-flow 输出的 P0 清单 | 审计清单质量 + 草拟逐条回答 |
| 需求 + 部分自答 | 补全缺口 + 标注待确认 |
| `@` 了相关原型/01/02 文档 | 优先采信文档，标注来源 |
| 附带截图 / 标注图 / 选中片段 | 先提取其中已确认项，再补缺口 |

**不触发**：用户已进入 02/03 方案阶段、仅咨询概念、或明确只要方案不要 P0。

## 与 pm-flow 的关系

- 本技能 = **预审与草拟**；`product-requirement-flow` + `product-manager` = **正式流程**。
- 草拟回答**不等于**用户确认；结尾须提示：请校对「❓ 待确认」项后，按编号粘贴给 pm-flow。
- **禁止**在本技能中输出 02 方案、03 功能清单或可交互原型。

## 交互知识增强（About Face 适配）

本技能在 P0 阶段不做视觉设计，但要提前识别会导致后续返工的交互问题。优先吸收以下原则：

1. **目标先于功能**：先确认用户想达成什么，不要被系统结构牵着走。
2. **贴近心智模型**：问题与草拟答应使用用户理解的对象与动作，不要泄露实现模型。
3. **姿态匹配**：
   - 独占姿态：可容纳更复杂的效率流
   - 短暂姿态：必须极简，用完即走
   - 后台姿态：尽量少打扰
   - 辅助姿态：兼顾轻量与连续操作
4. **导航可回退**：至少问清“我在哪里、我能去哪、我怎么回去”
5. **少打断，多可撤销**：优先问“是否可撤销/回退”，不要默认靠确认框兜底
6. **减少输入负担**：能默认、能猜、能复用就不要让用户多填
7. **沿用习惯用法**：已有交互模式能解决时，不轻易发明新模式
8. **控件与场景匹配**：少量互斥选项更适合单选；大量选项不该硬塞下拉；图标按钮若无文字/上下文易误解则要追问
9. **跨端差异显式确认**：Web、移动端、触控端的 Hover、点击精度、后退行为不能默认一致
10. **场景优先于问卷**：若用户目标、主路径和现有模块惯例已足以推出更优交互，不要把它继续挂成“待确认”

## 执行流程（按序）

### Step 1：解析输入

提取并记录：

- **需求摘要**（一句话，≤50 字）
- **模块/域**（如数据表格、图表、权限）
- **输入模式**（A 仅需求 / B 已有 P0 清单 / C 部分已答）
- **用户约束**（时间、平台、竞品对齐、@ 文件）
- **证据载体**（原话 / 粘贴文本 / 截图 / 图片标注 / @ 文件）
- **用户已明确表达的态度**（确认 / 反对 / 待定 / 建议）

#### Step 1.1：多模态证据提取（新增，必做）

若用户提供了截图、粘贴稿、红框、批注、下划线、圈选、箭头说明，先做以下动作再进入后续步骤：

1. **识别截图里被强调的对象**：如红框题号、截图中的某条 `P0-07`、某句“这里不对”。
2. **只提取用户实际表达过的立场**：
   - “应该点行直接开聊，点头像开名片” → 记为**已确认**
   - “感觉不对 / 牛头不对马嘴” → 记为**否定或需重写**
   - 仅红框但无明确结论 → 记为**关注项，不等于已确认**
3. **建立证据映射表**：`证据 -> 对应 P0 编号 -> 立场 -> 置信度`
4. **若截图只覆盖部分题目**，只能更新对应题，不得顺手改写其他题。

#### Step 1.2：已有问题与答案对齐检查（新增，必做）

当输入里已经存在 `P0-01...P0-NN` 的问题清单、待确认项、回复稿或其截图时，必须先做一次**对齐扫描**：

1. 问题是否是**开放题**还是**选项题**
2. 草拟回答是否真的回答了该题，而不是跳到实现建议
3. `待你确认` 中标记为待定的题，`可直接粘贴的回复稿` 中是否仍写成了肯定句
4. 若用户已在截图或原话中拍板，是否回写进摘要与回复稿

发现不一致时，必须在输出中显式指出：`P0-07 原为待定，但回复稿写成确定答案，需修正。`

### Step 2：检索上下文（必做）

在 `原型/模块/`、`原型/规范/` 下搜索同模块或同类能力：

1. 已有 `01-需求卡`、`02-需求方案`、`03-功能清单`
2. 相关 HTML 原型与 `04-其他` 竞品/分析文档
3. 同域参考（如「折线图」参考「柱状图」01）

检索后列出 **采信来源**（路径 + 用了哪几条）。无文档时注明「无同模块文档，按多维表通用惯例推断」。

#### Step 2.2：交互上下文判断（新增，必做）

若需求带有明显交互特征，必须补做以下判断：

1. **当前容器姿态**：这是独占页、弹框、抽屉、覆盖层、后台提醒，还是辅助面板？
2. **当前交互骨架**：用户主要在列表中切换、在表单中录入、在面板中配置，还是在消息流中快速完成动作？
3. **是否已有稳定习惯用法**：当前模块是否已经有一致的 Tab、筛选、返回、详情打开方式？
4. **是否存在跨端差异**：是否涉及移动端、触控、嵌入态、浏览器后退、刷新状态丢失？

若以上任一项会影响 P0 口径，必须在摘要或题表中体现，不得等到 02 方案才首次提及。

#### Step 2.3：交互默认可推断性判断（新增，必做）

对每个交互类问题，再做一次“是否真的需要问用户”的判断：

1. **用户目标是否单一明确**：例如“找人后立刻沟通”“删会话后快速找回群聊”
2. **主路径是否已给出最短动作链**：是否存在明显更顺手、更少一步的方案
3. **模块是否已有稳定惯例**：如列表点行打开详情、头像开名片、主按钮直接执行
4. **该取舍是否影响范围/权限/数据口径**：若不影响，只影响局部交互优先级，优先直接收敛
5. **是否存在明显反模式**：若某个备选会额外加一步、打断心流、增加认知负担，应降级为不推荐而不是继续平权提问

满足以下条件时，**直接给草拟答案，不列入待确认**：

- 与用户目标高度一致
- 符合当前模块或通用习惯用法
- 不改变范围、权限、数据归属
- 不会引发跨端/兼容/合规分歧

只有在以下情况才保留为 `待确认`：

- 不同选项会改变**范围、权限、数据口径、兼容边界**
- 现有证据互相冲突
- 两个方案在用户价值上都成立，且取舍取决于业务策略而非交互常识

#### Step 2.1：证据优先级（新增）

草拟回答时，证据优先级必须遵循：

1. **用户本轮明确原话 / 明确纠正**
2. **用户提供的截图、批注、选中片段**
3. **用户 @ 的原型 / 01 / 02 / 03 文档**
4. **模块现有真源文档与定稿原型**
5. **同类模块惯例推断**

高优先级证据一旦与低优先级冲突，必须以前者为准，并在摘要中说明冲突已被覆盖。

### Step 3：构建或审计 P0 问题

**五维框架**（与 pm-flow 一致；详见 [reference.md](reference.md)）：

1. 用户与场景
2. 对象与范围
3. 主路径
4. 规则与边界
5. 验收口径

**编号**：全局连续 `P0-01`～`P0-NN`（与 `.cursor/commands/pm-flow.md` 一致）；分组标题不占号。

**审计已有清单时**检查：

- 漏维、重复、过细（已到方案级）、过粗（无法验收）
- 与需求无关的题 → 标「建议删除」并说明
- 缺关键题 → 标「建议增补」并给出题干
- **问答不对题**：问题是选项题，回答却改成另一套开放结论
- **跨段不一致**：摘要 / 待确认 / 回复稿三处口径冲突
- **证据未落表**：截图或用户纠正已出现，但表格未体现
- **交互断层**：只问功能范围，没问入口、返回、定位、反馈、撤销
- **姿态失配**：把短暂/辅助场景问成重配置型工作流，或把后台提醒问成高打扰主流程
- **控件滥用**：少量选项却默认下拉；图标按钮无文字语义；需要直接操作却只给间接入口
- **过度上收确认**：本可由用户场景直接推断的交互默认，被机械列为“待确认”

**题量**：常规 8～15 条；小改 6～10；新模块 12～18。超出 18 条须合并或标 P1 澄清。

### Step 4：草拟回答

每条回答遵循：

| 标记 | 含义 |
|------|------|
| ✅ | 有文档/原型/用户原话依据 |
| 🔶 | 同类模块或产品惯例推断 |
| ❓ | 缺信息，必须用户拍板 |

**写作要求**：

- 每条 **≤80 字**；可验收、可判定，避免「尽量」「适当」
- **主路径** 3～7 步，箭头串联：`A → B → C`
- **范围** 分列「本期做 / 本期不做」（不做项 ≤5）
- **验收口径** 1～3 条，含可观测结果（步数、成功率、状态）
- 不确定写「待定」并说明缺什么信息

#### Step 4.1：选项题作答规则（新增，强制）

若原题是**选项题**（如 A/B/C/D、多选、二选一），草拟回答必须遵循以下规则：

1. **先对齐题型，再写答案**。不要把选项题偷换成自由发挥。
2. 草拟回答优先用以下格式之一：
   - `选 A：……`
   - `A + C：……`
   - `倾向 A，若 xxx 则改 B`
3. 若需要转写成自然语言，必须保证能**一一映射回选项**，并在依据里标明来源。
4. 若用户已在截图/原话里否定某个选项，不能在回复稿里继续输出该选项对应结论。
5. 若尚未拍板，回复稿必须写 `待定（A/B 二选一）`，不能冒充已确认。

#### Step 4.2：开放题作答规则（新增）

若原题是开放题，草拟回答应直接回答问题本身，不要把答案升级成方案：

- 问“点击成员主行为是什么” → 回答“点行开聊 / 点头像开名片”
- 不要写成“建议未来统一入口、减少心智负担”这类方案层描述

#### Step 4.4：交互型需求的草拟答约束（新增）

当题目涉及入口、列表、Tab、弹框、抽屉、面板、表单、筛选、状态反馈时，草拟回答还应满足：

1. **先回答主动作**：用户点哪里、看到什么、下一步去哪
2. **再回答反馈与回退**：默认态、空态、失败态、关闭/返回方式
3. **避免实现词汇**：少用“路由、实例、挂载、接口状态”这类实现模型词
4. **优先用户动作词**：打开、切换、返回、保存、撤销、重试、定位
5. **若存在明显交互风险**，宁可把它列入 `待你确认`，也不要在回复稿里假设通过
6. **若最佳交互已可推断**，应直接写结论 + 简短理由，不要为了“稳妥”继续把它挂成待确认

#### Step 4.5：交互题的直接作答规则（新增，强制）

当交互题可由场景直接推出时，按以下方式写：

1. **先下结论**：直接回答最合理的默认行为
2. **再补理由**：一句话说明它为何更符合场景
3. **不再伪装成开放选择题**：不要写成“是 A 还是 B？”后又不给判断

优先直接回答的典型场景：

- **找人后立即执行沟通**：默认点行直达主动作，不先绕路进名片
- **列表主对象操作**：默认点行打开主内容；头像/信息图标打开补充资料
- **短暂/辅助面板**：默认关闭成本低、返回路径清晰，不再额外弹确认
- **通知/系统消息**：默认轻打扰、快速定位，不扩成重流程配置

示例：

- 问：`成员点行后是直接开聊，还是先名片再发消息？`
- 若用户场景是“从通讯录快速找人沟通”，应直接答：
  `点成员行直接开聊；点头像打开名片。理由：主目标是立即沟通，名片属于补充信息，不应挡在主路径前。`

#### Step 4.6：跨段一致性约束（新增，强制）

生成第 2、3、4 段前，逐题检查：

1. `P0 表` 的状态是否与 `待你确认` 一致
2. `待你确认` 标为待定的题，`回复稿` 中是否也保留待定
3. 若用户已确认某题，三段里都应更新，不能只改其中一段
4. 摘要里的“已确认项 / 待确认项”数量是否与明细表一致

### Step 5：输出（固定结构）

按以下四段输出，便于复制：

```markdown
## 1. 预审摘要
- 需求理解：（一句话）
- 上下文：（引用的文档/原型，无则写无）
- 已确认输入：用户原话 / 截图 / 文档中已确定的题号
- 清单结论：题量 / 漏项 / 冗余 / 风险 / 已修正的不一致（≤5 条要点）

## 2. P0 澄清问题清单（含草拟答）
> 请校对 ❓ 项后，将「你的回答」列确认或修改，再按编号回复 pm-flow。

| 编号 | 维度 | 问题 | 草拟回答 | 状态 | 依据 |
|------|------|------|----------|------|------|
| P0-01 | 用户与场景 | … | … | 已确认/待确认 | ✅ 用户原话 / 🔶 推断 / ❓ 缺信息 |

## 3. 待你确认（仅 ❓ 与关键取舍）
1. …
2. …

## 4. 可直接粘贴的回复稿
P0-01：…
P0-02：…
…
```

**第 4 节**去掉标记与表格，只保留 `P0-NN：答案`，方便用户一键粘贴。

若某题仍待定，第 4 节必须保留 `待定` 字样，不得为了“看起来完整”擅自补成确定答案。

### Step 6：收尾

- 说明：校对后粘贴给 `/pm-flow` 或「一句话需求」对话即可进入 01/02/03。
- 若草拟中 ❓ 项 >3，建议用户先补答再开 pm-flow，避免方案返工。

## 质量自检（输出前必过）

- [ ] 五维均覆盖，或已说明某维 N/A 及理由
- [ ] 每题有草拟答或显式「待定」
- [ ] 主路径 ≤7 步，入口与完成态明确
- [ ] 有「本期不做」或等效边界
- [ ] 验收口径可观测，非空泛形容词
- [ ] 未越界输出方案对比/功能清单/原型
- [ ] 含「可直接粘贴的回复稿」
- [ ] 推断项（🔶）未冒充已确认事实
- [ ] 截图 / 红框 / 用户纠正已映射到对应 P0 编号
- [ ] 选项题答案能回指到原选项，不是自说自话
- [ ] `预审摘要` / `待你确认` / `回复稿` 三段无冲突
- [ ] 涉及交互时，已覆盖入口 / 完成态 / 返回方式 / 反馈状态中的关键项
- [ ] 涉及弹框/面板/通知时，已判断姿态是否匹配，不把轻场景问成重流程
- [ ] 涉及表单/输入时，已检查默认值、减少输入、即时反馈，而非只写校验规则
- [ ] 对交互题已判断“应直接回答”还是“必须确认”，未把可推断项机械塞进待确认

## 多维表领域默认假设（无文档时 🔶）

仅在无项目文档且用户未否定时使用，须在表中标注 🔶：

- **角色**：数据表编辑者 / 查看者；管理员配置权限
- **入口**：表内、视图内、项目编辑页、配置抽屉（按需求域选）
- **权限**：与现有模块权限体系一致；无权限则入口不可见或禁用+提示
- **隐性态**：加载、空状态、错误提示、无权限需显式列入规则与边界或验收
- **命名**：mock 人名用西游记角色（项目规范）
