# P0 预审样例

## 样例 A：仅一句话需求 → 生成问题 + 草拟答

**用户输入**

```
预审 P0：一句话需求: 数据表格列头支持快捷筛选，像 Excel 那样点列头就能筛
```

**输出节选（结构示意）**

### 1. 预审摘要

- **需求理解**：表格视图列头增加快捷筛选，降低筛选操作步数。
- **上下文**：`原型/模块/数据表格/列头快捷筛选/方案/01-需求卡-列头快捷筛选.md`（✅）
- **清单结论**：12 题；无漏维；建议删除原「配色」类过细题 0 条；风险：与全局筛选面板边界需确认。

### 2. P0 澄清问题清单（节选）

| 编号 | 维度 | 问题 | 草拟回答 | 状态 | 依据 |
|------|------|------|----------|------|------|
| P0-01 | 用户与场景 | 主用户与触发时机？ | 表格编辑者；浏览/核对数据时需按列快速缩小可见行 | 已确认 | ✅ |
| P0-02 | 对象与范围 | 筛选作用对象与范围？ | 当前视图可见列；仅影响当前视图展示，不改底层权限 | 待确认 | 🔶 |
| P0-03 | 对象与范围 | 本期做 / 本期不做？ | **In**：列头唤起筛选、条件应用、清除筛选；**Out**：跨表筛选、保存为视图 | 已确认 | ✅ |
| P0-04 | 主路径 | 主路径步骤？ | 点列头筛选图标 → 选条件/勾选值 → 应用 → 列表刷新 | 已确认 | ✅ |
| P0-05 | 规则与边界 | 无权限列的表现？ | 无读权限列不展示筛选入口 | 待确认 | 🔶 |
| P0-06 | 规则与边界 | 空值/枚举过多？ | 空值单独项；枚举超上限时仅搜索+前 N 条或转条件筛选 | 已确认 | ✅ |
| P0-07 | 验收口径 | 成功标准？ | ① 3 步内完成单列筛选 ② 筛选后行集与条件一致 ③ 清除后恢复 | 已确认 | ✅ |

### 3. 待你确认

1. 是否与工具栏「筛选」面板并存？并存时以谁为准？（P0-02 延伸）
2. 移动端是否本期覆盖？（文档未写 → ❓）

### 4. 可直接粘贴的回复稿

```
P0-01：表格编辑者；在表格视图浏览数据时需按列快速缩小可见行集。
P0-02：作用于当前视图列；仅影响展示，不改变行级权限。
…
```

## 样例 C：截图纠错 + 选项题对齐

**用户输入**

```markdown
/pm-p0-preflight

这是 pm-flow 的一轮结果截图，我觉得不太对：
- 红框的是 `P0-07`
- 我真正想表达的是：**点击成员主行为应为“点行直接开聊，点头像开名片”**
- 另外这套问题很多是选项题，但回复稿没有按选项回答
```

**截图中的关键信息**

- 红框定位在 `P0-07`
- `待你确认` 里仍写 `P0-07 成员主行为`
- `可直接粘贴的回复稿` 里却已经写成了另一版确定答案

**正确输出要点**

### 1. 预审摘要

- **需求理解**：用户不是要重做方案，而是要**纠正上一轮 P0 预审的失配**。
- **已确认输入**：截图已确认 `P0-07`；用户原话确认“点行直接开聊，点头像开名片”。
- **清单结论**：
  - `P0-07` 存在**跨段不一致**
  - 原问题中有选项题，但回复稿未按选项映射
  - 本轮重点是**修正已有问答**，不是扩写新方案

### 2. P0 澄清问题清单（节选）

| 编号 | 维度 | 问题 | 草拟回答 | 状态 | 依据 |
|------|------|------|----------|------|------|
| P0-07 | 主路径 | 点击组织成员后的主行为？ | 点成员行：直接开聊；点成员头像：打开名片 | 已确认 | ✅ 用户原话 + ✅ 截图 |
| P0-08 | 规则与边界 | 成员名片是否仍保留“发消息”按钮？ | 保留；名片作为补充路径，不替代点行直聊 | 待确认 | 🔶 |

### 3. 待你确认

1. `P0-08` 是否保留名片底部“发消息”按钮？
2. 其他选项题是否也要统一改写成 `选 A / 选 B` 形式？

### 4. 可直接粘贴的回复稿

```markdown
P0-07：点击成员行直接开聊；点击成员头像打开名片。
P0-08：待定，倾向保留名片中的“发消息”按钮作为补充入口。
```

**这个样例要说明的事**

- 截图中的红框 = 重点修正对象，不等于所有题都已确认
- 用户原话的优先级高于旧草稿
- `待你确认` 和 `回复稿` 必须一致，不能一边待定、一边写死
- 开放题要直接回答问题，不要升级成方案建议
- 若用户场景已足够明确，像 `P0-07` 这类交互题应**直接收敛为答案**，不应继续挂成待确认

## 样例 E：交互型需求 → 补问入口、返回、姿态

**用户输入**

```markdown
/pm-p0-preflight
一句话需求：在聊天会话列表增加“通讯录”入口，点开后能看到群聊和组织成员
```

**好的预审，不应只问“有哪些人/群”**

还应主动补以下交互问题：

1. 这是**短暂入口**还是支持连续浏览的辅助面板？
2. 通讯录的**主入口**在哪，和搜索栏、「+」的关系是什么？
3. 打开后是覆盖左侧列表、替换整个区域，还是抽屉展开？
4. 用户点成员/点群后会发生什么？如何关闭、如何回到原列表？
5. 是否有空态、加载失败、搜索无结果？

**输出节选（结构示意）**

| 编号 | 维度 | 问题 | 草拟回答 | 状态 | 依据 |
|------|------|------|----------|------|------|
| P0-04 | 主路径 | 通讯录入口放在哪里？ | 选 A：会话列表底部固定入口 | 待确认 | 🔶 |
| P0-05 | 主路径 | 打开后以什么形态承载？ | 倾向 A：覆盖左侧会话列表，不替换右侧聊天区 | 待确认 | 🔶 |
| P0-06 | 主路径 | 点击成员与点击群聊后的主行为？ | 点群直接进群；点成员行开聊，点头像开名片 | 待确认 | ✅/🔶 |
| P0-08 | 规则与边界 | 面板如何关闭或返回？ | 面板右上关闭；再次点“通讯录”也可返回 | 待确认 | 🔶 |
| P0-09 | 规则与边界 | 空态/加载失败怎么表现？ | 无群、无成员、无搜索结果、失败重试均需明确 | 已确认 | 🔶 |

**这个样例要说明的事**

- 交互型需求不能只问对象和范围，还要问**入口、姿态、回退、反馈**
- P0 阶段就要识别“这是短暂姿态还是辅助姿态”，否则 02 很容易返工
- 回答应优先使用用户动作词，而不是实现词

## 样例 F：用户场景已明确 → 交互题直接回答

**用户输入**

```markdown
/pm-p0-preflight
一句话需求：在聊天通讯录里，用户找到成员后要尽快发起沟通
补充：这个入口主要是为了“快速找回人并开聊”，不是看详细资料
```

**错误写法（不要这样）**

- `P0-13：成员点行后是“直接开聊”还是“先名片再发消息”？（草拟：直接开聊；若你更倾向名片优先请确认）`

问题在于：

- 用户目标已经明确是“尽快发起沟通”
- “先名片再发消息”明显多一步，且把补充路径抬成了主路径
- 这类题继续挂待确认，会让 P0 变成无差别问卷

**正确写法**

| 编号 | 维度 | 问题 | 草拟回答 | 状态 | 依据 |
|------|------|------|----------|------|------|
| P0-13 | 主路径 | 点击成员后的默认行为？ | 点成员行直接开聊；点头像打开名片。理由：主目标是快速沟通，名片是补充路径。 | 已确认 | ✅ 用户场景 + 🔶 模块惯例 |

**可直接粘贴的回复稿**

```markdown
P0-13：点成员行直接开聊；点头像打开名片。主目标是快速沟通，名片作为补充路径。
```

**这个样例要说明的事**

- 不是所有交互题都要上收确认
- 当用户目标、主路径、模块惯例都一致时，skill 应主动收敛，而不是把判断甩回给用户
