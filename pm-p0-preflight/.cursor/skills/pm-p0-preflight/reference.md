# P0 澄清：标准题库与审计细则

供 `pm-p0-preflight` 生成/审计问题时选用。不是每题必问——按需求裁剪，避免问卷式堆砌。

---

## 一、五维框架与 01 需求卡映射

| P0 维度 | 01 需求卡章节 | 核心产出 |
|---------|---------------|----------|
| 用户与场景 | §3 Who/When | 主角色、触发时机、频率、替代方案 |
| 对象与范围 | §5 Scope | 本期做 / 本期不做、影响面 |
| 主路径 | §6 Happy Path | 3～7 步操作链 + 入口/完成态 |
| 规则与边界 | §7 Rules & Edges | 权限、校验、空/加载/错误、撤销、性能 |
| 验收口径 | §4 成功标准 | 1～3 条可观测结果 |

## 二、标准题库（按维）

### 1. 用户与场景

| ID | 问题模板 | 何时必问 |
|----|----------|----------|
| U1 | 主用户是谁？在主路径里承担什么动作？ | 总是 |
| U2 | 在什么页面/模块、什么时机触发？频率？ | 总是 |
| U3 | 是否有次角色（只读、审批、管理员）？差异？ | 涉及权限/协作 |
| U4 | 用户今天怎么解决？新方案替换成本？ | 新能力/改版 |
| U5 | 典型业务场景 1 句话（谁、何时、要达成什么）？ | 新模块 |

### 2. 对象与范围

| ID | 问题模板 | 何时必问 |
|----|----------|----------|
| S1 | 操作/配置的对象是什么（字段、视图、组件、记录）？ | 总是 |
| S2 | 影响范围：仅当前表/视图/页，还是项目/空间级？ | 总是 |
| S3 | 本期做（In）最小闭环是什么？ | 总是 |
| S4 | 本期明确不做（Out）有哪些？ | 总是 |
| S5 | 与相邻模块边界（如视图 vs 数据源 vs 权限）？ | 跨模块 |
| S6 | 是否兼容旧数据/旧配置？迁移要求？ | 改版/字段类 |

### 3. 主路径

| ID | 问题模板 | 何时必问 |
|----|----------|----------|
| P1 | 用户从哪个入口进入？完成态是什么？ | 总是 |
| P2 | 主路径逐步操作（3～7 步）？ | 总是 |
| P3 | 是否有分支路径（配置态 vs 使用态）？ | 双态能力 |
| P4 | 与同类已有能力是否一致（如柱状图 vs 折线图）？ | 扩展类 |
| P5 | 误操作后如何返回/取消？ | 有破坏性操作 |

### 4. 规则与边界

| ID | 问题模板 | 何时必问 |
|----|----------|----------|
| R1 | 谁可见/可操作？无权限时表现？ | 总是 |
| R2 | 输入/配置校验与限制？ | 有表单/配置 |
| R3 | 空状态、加载态、失败提示？ | 总是 |
| R4 | 是否可撤销/回滚？ | 写操作 |
| R5 | 数据量/性能边界（行数、层级、并发）？ | 列表/树/导入 |
| R6 | 多终端（Web/移动/嵌入）差异？ | 跨端 |
| R7 | 与竞品或内部参照对齐点？ | 用户提及对标 |

### 5. 验收口径

| ID | 问题模板 | 何时必问 |
|----|----------|----------|
| A1 | 成功标准 1～3 条（可量化或可观测）？ | 总是 |
| A2 | 主路径几步内完成？ | 有效率要求时 |
| A3 | 必须覆盖的异常/边界验收点？ | 有高风险写操作 |
| A4 | 过程指标或灰度标准（可选）？ | 大需求 |

## 三、清单审计细则

### 3.1 漏项信号

| 信号 | 建议增补 |
|------|----------|
| 有写无读 / 有增无删 | CRUD 边界题（S4/A3） |
| 有配置无使用态 | P3 双路径 |
| 有功能无权限 | R1 |
| 有交互无失败态 | R3 |
| 无「本期不做」 | S4 |
| 验收不可观测 | 重写 A1（含步数/状态/提示文案） |
| 用户已给截图/批注却未落到题号 | 增加“证据映射”检查 |
| `待确认` 与 `回复稿` 口径不同 | 增加“一致性修正” |
| 只写功能，不写入口/返回/反馈 | 增补交互骨架题（P1/P2/R3） |
| 有弹框/面板/通知，但未说明是否打断主流程 | 增补姿态与打扰度问题 |

### 3.2 冗余信号

| 信号 | 处理 |
|------|------|
| 问到具体控件像素/色值 | 标 P1 或删除，属原型阶段 |
| 问到技术选型 | 标「方案阶段」，移出 P0 |
| 与需求无关角色 | 删除 |
| 同一信息问两遍 | 合并 |
| 问题是选项题，回答却绕开选项 | 重写答案，先回到题型 |
| 在 P0 阶段追问像素/动画细节 | 后移到原型或交互细化阶段 |

### 3.3 粒度信号

| 过粗 | 过细（应后移） |
|------|----------------|
| 「体验要好」 | 「按钮圆角 8px」 |
| 「支持导出」 | 「导出用 SheetJS 还是后端」 |
| 「权限控制」 | 「JWT 还是 Session」 |

### 3.6 交互坏味道（新增）

| 信号 | 处理 |
|------|------|
| 入口过多但无主入口 | 增问“首选入口/默认路径” |
| 用户进入后不知道自己在哪 | 增问“当前态、选中态、返回路径” |
| 功能靠模态框层层套模态框 | 追问是否可改抽屉/面板/非模态路径 |
| 用确认框代替防错或撤销 | 追问“是否可撤销/预防误操作” |
| 需要频繁填写重复信息 | 追问默认值、自动带入、复用历史 |
| 图标按钮承担关键动作却无文字语义 | 追问是否需文字、标签或更直接入口 |
| 少量互斥选项却用下拉 | 追问是否应改单选或显式切换 |
| 移动端沿用 Hover / Tooltips 逻辑 | 追问触控替代方案 |
| 用户主目标很明确，却把最佳默认交互继续挂成“待确认” | 直接收敛为草拟答，并给出场景理由 |

### 3.4 一致性信号（新增）

| 信号 | 处理 |
|------|------|
| 摘要说“待定”，回复稿写成肯定句 | 以“待定”为准，回滚回复稿 |
| 待确认列了某题，P0 表状态却写已确认 | 重新判定状态，并同步三段 |
| 用户截图已拍板，但摘要仍写待定 | 用截图证据更新摘要与回复稿 |
| 同一题在三段里用了不同口径 | 保留最高优先级证据对应口径，其他全部改齐 |

### 3.5 证据优先级（新增）

从高到低：

1. 用户本轮明确原话/纠正
2. 用户截图中的文字、红框、批注
3. 用户 @ 的文档或当前粘贴内容
4. 项目真源文档、定稿原型
5. 同类惯例推断

低优先级内容不得覆盖高优先级内容。

## 四、草拟答写法规范

### 4.1 主路径

```
入口 → 步骤2 → 步骤3 → … → 完成态（≤7 步）
```

- 动词+宾语；不用「处理」「支持」「管理」
- 示例：`打开添加组件抽屉 → 选折线图 → 落位 → 自动打开配置抽屉 → 绑定数据源 → 出图`

### 4.2 验收口径

每条含：**前置 + 操作 + 可判定结果**

- ✅ `未绑数据源时落位后 1 秒内自动打开配置抽屉`
- ❌ `配置体验流畅`

### 4.3 本期不做

- 每条 ≤30 字；说明「不做」而非「下期做」
- 与主路径无交集；避免把 P0 能力标成 Out

### 4.4 选项题写法（新增）

若原题是选项题，优先写成：

- `选 A：xxx`
- `选 B + C：xxx`
- `待定（A/B 二选一，取决于 xxx）`

禁止写法：

- 原题问 A/B/C，回答却完全不出现 A/B/C 映射
- 原题未确认，回复稿却直接替用户定案

### 4.5 截图/批注输入写法（新增）

若用户通过截图表达意见：

- 先在摘要中写明：`截图已确认：P0-07`
- 再在表格对应行落具体结论
- 最后把同一结论同步到可粘贴回复稿

若截图只是圈出问题、未表达结论：

- 标为“关注项”，不要误写成已确认

### 4.6 交互类问题优先模板（新增）

当需求明显是交互型而非纯数据型时，优先从下列模板里挑题，而不是只问范围和权限：

#### 入口与姿态

- 这是独占页、短暂弹层、辅助面板，还是后台通知？
- 用户的**首选入口**是什么？其他入口是补充还是替代？
- 该交互应该“用完即走”，还是支持连续深入操作？

#### 主路径与回退

- 用户进入后第一眼看到什么？默认落在哪个状态？
- 主动作是什么？点完后的即时反馈是什么？
- 用户如何返回上一步、关闭当前层、回到来源位置？
- 哪个动作是**主路径默认**，哪个只是补充路径？

#### 反馈与防错

- 是否有空态、加载态、失败态、重试？
- 关键动作是靠确认框兜底，还是靠禁用/预防/撤销？
- 是否能避免重复输入？哪些值可默认带入？

#### 控件与跨端

- 这个场景为何用 Tab / 下拉 / 单选 / 开关？是否符合用户习惯？
- 图标按钮是否需要文字或更直白的入口？
- Web、移动端、触控端是否有 Hover、返回、点击精度差异？

### 4.7 交互类草拟答标准（新增）

交互类回答优先写成：

`入口 + 用户动作 + 系统反馈 + 返回方式`

示例：

- `点会话列表底部“通讯录” → 左侧覆盖通讯录面板 → 点成员行直接开聊 → 点关闭返回会话列表`
- `切换到“待办”Tab → 默认展示未完成项 → 勾选后即时沉底，并提供撤销入口`

### 4.8 何时直接回答，不再上收确认（新增）

满足以下 4 条中的至少 3 条，可直接作为草拟答，不列入待确认：

1. 用户目标单一明确
2. 最短路径明显优于备选
3. 与现有模块惯例一致
4. 不影响范围、权限、数据口径

典型例子：

- 用户目标是“快速找到人并发起沟通”
- 备选 A=`点行直接开聊`
- 备选 B=`先开名片再点发消息`

此时应默认采用 A，并补一句理由：

`点行直接开聊；点头像打开名片。理由：主目标是快速沟通，先名片会增加一步，不符合主路径最短原则。`

## 五、需求规模 → 建议题量

| 类型 | 特征 | 建议 P0 题量 |
|------|------|--------------|
| 小 | 单入口、同模块扩展、有现成参照 | 6～10 |
| 中 | 新交互、双态、权限相关 | 10～14 |
| 大 | 新模块、跨团队、数据迁移 | 14～18 |
